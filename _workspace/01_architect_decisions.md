# B2B Tennis 복식 리그 - 설계 결정 사항

## 1. 시즌 테이블 제거

**결정:** seasons 테이블을 두지 않는다. 기간 필터링은 API 쿼리 파라미터(from, to)로 처리한다.

**근거:**
- 요구사항에 시즌 개념이 명시되지 않았다.
- 요구사항에 "조회 기간 선택: 특정 날짜 ~ 특정 날짜"로 명시되어 있다.
- 시즌 테이블은 관리 오버헤드(시즌 생성/마감 UI)를 추가하며, 20명 내외 소규모 리그에 불필요하다.
- 향후 필요 시 seasons 테이블을 추가하고 games에 season_id FK를 붙이면 된다.

---

## 2. 복식 팀 모델링 (game_teams)

**결정:** 별도의 teams 테이블 없이 game_teams에 player1_id, player2_id를 직접 저장한다.

**근거:**
- 매 경기마다 팀 구성이 달라질 수 있다 (고정 팀이 아님).
- 고정 teams 테이블을 두면 매번 새로운 조합이 생길 때 팀을 생성해야 하므로 관리가 복잡해진다.
- game_teams 테이블에 직접 2인의 ID를 저장하면 구조가 단순하고 쿼리도 직관적이다.
- player1_id < player2_id 같은 정규화는 하지 않는다. 입력 순서를 보존하는 것이 UI 표현에 유리하다.

---

## 3. 같은 선수 양팀 등록 방지 - API 레벨 검증

**결정:** DB CHECK 제약 대신 API 레벨에서 4명의 선수 ID가 모두 다른지 검증한다.

**근거:**
- DB의 CHECK 제약으로는 cross-row 비교(home 팀과 away 팀 간)를 처리하기 어렵다.
- 트리거로 처리할 수 있으나 에러 메시지가 불친절하고 디버깅이 어렵다.
- API 레벨에서 검증하면 사용자에게 명확한 에러 메시지(예: "선수 A가 양팀에 등록되어 있습니다")를 반환할 수 있다.
- 같은 팀 내 중복(player1_id = player2_id)은 DB CHECK로도 처리한다 (이중 방어).

---

## 4. Soft Delete 전략 (profiles.is_active)

**결정:** 선수 삭제 시 경기 기록이 있으면 `is_active = false`로 처리한다. 경기 기록이 없으면 hard delete한다.

**근거:**
- 경기 기록에 FK로 연결된 선수를 삭제하면 데이터 정합성이 깨진다.
- ON DELETE SET NULL을 쓸 수도 있으나, "누가 경기했는지" 정보가 사라지므로 부적절하다.
- is_active = false인 선수는 선수 목록에서 기본적으로 제외하되, 과거 경기 기록에서는 이름이 표시된다.
- 경기 기록이 없는 선수는 FK 참조가 없으므로 안전하게 hard delete 가능하다.

---

## 5. sets_won / games_won 비정규화

**결정:** game_teams 테이블에 sets_won, games_won을 별도 컬럼으로 저장한다 (game_sets에서 계산 가능한 값임에도).

**근거:**
- 순위 계산 쿼리에서 매번 game_sets를 JOIN하여 집계하면 성능이 저하된다.
- 경기 기록 시 game_sets 입력과 동시에 game_teams의 집계 값을 계산하여 저장한다.
- 이 값들은 경기 기록 시점에 한 번 계산되며 이후 변경되지 않는다.
- 데이터 불일치 위험: 경기 기록 API에서 트랜잭션으로 일괄 처리하여 방지한다.

---

## 6. RLS 정책 설계

**결정:**
- 읽기(SELECT): 인증된 사용자 전원에게 허용.
- 쓰기(INSERT): 인증된 사용자 전원에게 허용. 단, games의 recorded_by는 자신의 ID만 가능.
- 삭제(DELETE): games는 recorded_by가 자신인 경우만. game_teams, game_sets는 CASCADE.
- profiles 수정: 본인 레코드만.

**근거:**
- 요구사항: "관리자 구분 없이 모든 회원이 경기 기록 입력 가능."
- 소규모 리그(20명 내외)이므로 복잡한 역할 기반 접근 제어가 불필요하다.
- 경기 삭제는 기록자만 가능하게 하여 실수로 다른 사람의 기록을 삭제하는 것을 방지한다.
- profiles은 본인만 수정 가능하게 하여 닉네임 등이 타인에 의해 변경되는 것을 방지한다.

---

## 7. profiles와 auth.users 1:1 관계 + Auth 트리거

**결정:** profiles.id = auth.users.id (UUID 동일). 신규 가입 시 트리거로 profiles 행을 자동 생성한다.

**근거:**
- Supabase 공식 패턴이다.
- auth.users 테이블은 Supabase가 관리하므로 직접 접근하지 않고, public.profiles로 확장 필드를 관리한다.
- 트리거로 자동 생성하면 가입과 프로필 생성의 원자성이 보장된다.
- name은 `raw_user_meta_data->>'name'` 또는 email에서 가져온다.

---

## 8. 세트 점수 유효성 검증 범위

**결정:** 정식 테니스 점수 규칙(6-x, 7-5, 7-6)을 기본으로 하되, 동호회 관행(4게임제 등)은 초기 버전에서 미지원한다.

**근거:**
- 요구사항 예시가 "6-4, 7-5, 6-3"으로 정식 규칙 기반이다.
- 정식 규칙만 검증하면 로직이 단순하고 오입력을 방지할 수 있다.
- 향후 "자유 점수 모드" 옵션을 추가하면 확장 가능하다.
- 타이브레이크 점수 검증: 7-6일 때 home_tiebreak, away_tiebreak 필수. 나머지는 null.

---

## 9. 순위 계산 방식 - 실시간 쿼리

**결정:** 순위 조회 시 실시간으로 SQL 집계 쿼리를 실행한다. Materialized View나 캐시 테이블은 사용하지 않는다.

**근거:**
- 20명 내외, 경기 수가 많지 않으므로 실시간 집계의 성능 부담이 미미하다.
- game_teams에 sets_won, games_won 비정규화 컬럼이 있어 game_sets JOIN 없이 집계 가능하다.
- Materialized View는 갱신 타이밍 관리가 필요하고 복잡도가 증가한다.
- 향후 사용자 수 증가 시 Supabase RPC(stored function)로 전환하면 된다.

---

## 10. 경기 상태(status) 단순화

**결정:** games.status는 `completed`와 `cancelled` 두 가지만 사용한다. `scheduled` 상태는 제외한다.

**근거:**
- 요구사항에 경기 예약(scheduling) 기능이 없다.
- 경기 기록은 이미 완료된 결과를 입력하는 것이다.
- `cancelled`는 삭제 대신 취소 처리가 필요할 때를 위한 예비 상태이다.
- 초기 버전에서는 삭제(hard delete)로 충분하지만, 감사 추적이 필요할 때 cancelled로 전환 가능하다.

---

## 11. API 라우트 구조 (Next.js App Router)

**결정:** Next.js Route Handlers(`app/api/.../route.ts`)를 사용한다. Server Actions는 경기 기록 등 폼 처리에 보조적으로 사용 가능하나, API 명세의 주 구현체는 Route Handlers이다.

**근거:**
- Route Handlers는 REST API 형태로 명확한 엔드포인트 구조를 제공한다.
- 향후 모바일 앱 등 외부 클라이언트 지원 시 API를 그대로 활용 가능하다.
- Server Actions는 Next.js 내부에서만 사용 가능하므로 API 명세 대상으로는 부적합하다.

---

## 12. 페이지네이션

**결정:** 경기 목록에 오프셋 기반 페이지네이션을 적용한다. 순위, 선수 목록은 페이지네이션 없이 전체 반환한다.

**근거:**
- 경기 목록은 누적되므로 페이지네이션이 필요하다.
- 선수 목록: 최대 20명 내외로 전체 반환해도 문제없다.
- 순위: 기간 필터링 결과이므로 전체 반환이 적절하다.
- 커서 기반 대신 오프셋 기반을 선택한 이유: 소규모 데이터이고, 페이지 번호 기반 UI가 직관적이다.
